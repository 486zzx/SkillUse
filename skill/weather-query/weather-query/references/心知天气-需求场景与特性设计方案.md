# 心知天气：需求场景与特性设计方案

本文档基于 [心知天气 API 使用手册（V4版）· 语雀](https://seniverse.yuque.com/hyper_data/api_v4) 及 [心知天气官方文档](https://docs.seniverse.com/) 整理，从需求场景出发给出「基于心知天气 API」的天气查询能力设计与扩展方案。语雀页面抓取超时，需求与特性设计参考官方接口能力与常见业务场景归纳得出。

---

## 一、需求场景

### 1.1 按接口能力映射的场景

| 场景 | 用户表述示例 | 心知接口 | 说明 |
|------|--------------|----------|------|
| 当前天气 | 「北京现在多少度」「上海天气怎么样」 | 天气实况 now | 温度、体感、现象、风力；免费仅 3 项。 |
| 今日/明日/多日 | 「今天会下雨吗」「未来一周天气」 | 逐日预报 daily | start=0/1，days=3~15（免费 3 天）。 |
| 逐小时 | 「下午三点会不会下雨」 | 24h 逐时 hourly / 15 天逐 3h hourly3h | 精细化时段。 |
| 生活与穿衣 | 「该穿什么」「紫外线强不强」 | 生活指数 suggestion | 仅中国城市；免费 6 项基本类。 |
| 空气质量 | 「北京空气质量」「PM2.5」 | 空气质量实况 air/now | AQI、PM2.5、PM10、首要污染物等。 |
| 地点解析 | 「查一下苏州天气」 | 城市搜索 search + location | 中文/拼音/IP→城市 ID/名称，供后续接口用。 |
| 昨日/历史日 | 「昨天天气」 | 逐日 daily | start=-1 昨天，start=-2 前天；或 start=yyyy/m/d。 |

### 1.2 用户角色与诉求

| 角色 | 诉求 | 心知能力对应 |
|------|------|--------------|
| 普通 C 端用户 | 当前温度、今明两天、是否下雨、穿衣建议 | now + daily + suggestion |
| 出行/旅行 | 多日预报、降水、生活指数 | daily + suggestion |
| 健康/敏感人群 | 空气质量、紫外线、运动指数 | air/now + suggestion |
| 企业/应用 | 多城市、稳定、可签名 | 全接口 + 签名验证 |
| 精细化场景 | 逐时、逐 3 小时 | hourly + hourly3h |

### 1.3 输入形态与约束

- **地点**：城市 ID、中文名、拼音/英文、经纬度（纬度:经度）、IP；需在调用前统一为接口支持的 location 形式。
- **时间**：今天(0)、明天(1)、昨天(-1)、前天(-2)、或具体日期 yyyy/m/d；逐日用 start + days。
- **语言**：zh-Hans、en、ja 等，与返回文案一致。
- **单位**：c（摄氏/km/h）或 f（华氏/mph）。
- **限制**：生活指数仅中国；免费版字段/天数受限；需 API Key（或公钥+签名）。

## 二、特性设计方案

### 2.1 与现有 skill 的整合方式

当前 weatherPrj 内天气 skill 以 **Open-Meteo** 为主（无 KEY、全球）。心知天气可作为**可选数据源**：

- **方案 A**：独立「心知天气」skill，入口与脚本单独（如 `run_seniverse_weather.py`），与 Open-Meteo 并列，由调用方或配置选择。
- **方案 B**：统一入口 `run_weather_search.py` 通过环境变量或参数选择 provider（open-meteo / seniverse），内部路由到不同实现。

以下特性按「心知作为独立或可切换数据源」设计。

### 2.2 基础能力（P0）

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| S1 | 心知天气实况查询 | 当前天气 | 脚本/模块：接收 location、language、unit；调 `/v3/weather/now.json`；输出统一 JSON（地点、温度、体感、现象、风力等）；免费版只展示有返回的 3 项。 |
| S2 | 心知逐日预报 | 今日/明日/多日 | 调 `/v3/weather/daily.json`，start=0/1 或日期，days=3~15；输出日列表（日期、最高/最低温、现象、降水等）；区分免费 3 天与付费 15 天。 |
| S3 | 地点标准化 | 中文/拼音/IP→location | 优先用城市搜索接口或本地 city_map 得到城市 ID/标准名，再作为 location 调用 now/daily；支持「北京」「beijing」、经纬度、IP。 |
| S4 | 认证配置 | 所有请求 | 支持环境变量 `SENIVERSE_KEY`（私钥）或 `SENIVERSE_UID`+`SENIVERSE_PRIVATE_KEY`（公钥+私钥做签名）；脚本内统一封装请求与签名逻辑。 |

### 2.3 增强能力（P1）

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| S5 | 生活指数 | 穿衣、紫外线、运动 | 调 `/v3/life/suggestion.json`；仅当 location 为中国城市时调用；免费 6 项、付费 27 项；与实况/逐日合并输出或单独接口。 |
| S6 | 空气质量 | AQI、PM2.5 | 调 `/v3/air/now.json`；可选与实况同次请求合并展示；输出 AQI、PM2.5、PM10、首要污染物、质量等级。 |
| S7 | 24 小时逐时 | 未来半天到一天 | 调 `/v3/weather/hourly.json`；可选参数控制是否返回逐时；与 daily 互补。 |
| S8 | 15 天逐 3 小时 | 精细化时段 | 调 `/v3/weather/hourly3h.json`；返回 3h/6h/12h 分段数据；适合「几点下雨」类需求。 |
| S9 | 昨日/历史某日 | 昨天、前天、某日 | daily 接口 start=-1/-2 或 start=yyyy/m/d，days=1；输出单日或少量日；用于对比或回顾。 |

### 2.4 扩展能力（P2）

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| S10 | 签名验证 | 安全与生产 | 使用 uid+ts+ttl+sig，私钥仅存服务端；在封装层实现 HMAC-SHA1+Base64+URLEncode。 |
| S11 | 多语言与单位 | 国际化 | 请求参数 language、unit 可配置；输出保留单位标识，便于前端展示。 |
| S12 | 城市搜索独立接口 | 地点补全、选城市 | 暴露「仅城市搜索」脚本或 API，返回城市 ID/名称列表，供上游或人工选择后再查天气。 |
| S13 | 海洋/农业/图层 | 行业场景 | 按心知文档中的海洋类、农业类、气象图层等接口，按需增加脚本或文档说明，优先级低于通用天气与生活指数。 |

### 2.5 脚本与数据流设计

```
用户 / 模型
    ↓
统一入口（按 provider 分流）
    ├── Open-Meteo：run_weather_search.py [现有]
    └── 心知天气：run_seniverse_weather.py [新增]
            ├── 认证：SENIVERSE_KEY 或 uid+sig
            ├── 地点：search / city_map → location
            └── 按需调用
                ├── /v3/weather/now.json          → 实况
                ├── /v3/weather/daily.json       → 逐日
                ├── /v3/weather/hourly.json      → 24h 逐时（可选）
                ├── /v3/weather/hourly3h.json     → 15 天逐 3h（可选）
                ├── /v3/air/now.json             → 空气质量（可选）
                └── /v3/life/suggestion.json     → 生活指数（可选，仅中国）
    ↓
统一输出结构（便于模型/前端）
    location, current, daily, hourly（可选）, air（可选）, suggestion（可选）
```

### 2.6 与 Open-Meteo 的对比与选型建议

| 维度 | 心知天气 | Open-Meteo |
|------|----------|------------|
| 认证 | 必须 Key 或签名 | 非商业无需 Key |
| 中文支持 | 原生中文、中国城市与生活指数 | 需本地 city_map 映射 |
| 生活指数 | 27 项、仅中国 | 无独立生活指数（可自算 UV 等） |
| 空气质量 | 有独立 air/now | 有独立 API |
| 历史 | 仅昨日/前天/指定日（daily start） | 1940 年至今 archive |
| 海洋/网格 | 有海洋、农业、图层等 | 有海洋 API |

**选型建议**：国内产品、需要生活指数与中文体验时优先心知；无 Key、全球或历史长序列时用 Open-Meteo；也可双源并存由配置切换。

### 2.7 错误与边界

| 情况 | 处理 |
|------|------|
| Key 未配置或无效 | 明确提示配置 SENIVERSE_KEY 或签名参数。 |
| location 无法解析 | 先尝试城市搜索或 city_map；失败则返回「无法解析地点」。 |
| 生活指数非中国城市 | 不调 suggestion 或返回「仅支持中国城市」。 |
| 免费版天数/字段限制 | 文档与输出中说明「免费 3 天」「实况仅 3 项」等，避免用户误解。 |
| 限流/配额 | 捕获 HTTP 4xx/5xx，记录日志并返回「服务暂时不可用或已达调用上限」。 |

## 三、实施优先级建议

1. **第一阶段**：S1 实况 + S2 逐日 + S3 地点 + S4 认证，实现心知最小可用闭环（可与现有 Open-Meteo 入口并列）。
2. **第二阶段**：S5 生活指数 + S6 空气质量，提升国内场景体验。
3. **第三阶段**：S7/S8 逐时与逐 3h、S9 昨日/历史日、S10 签名验证，按需上线。
4. **第四阶段**：S12 城市搜索独立、S13 海洋/农业/图层，按业务需要扩展。
