# 列车班次查询

根据用户自然语言查询列车班次与车票：从用户输入中提取出发站、到达站、**出发时间**（自然语言，如「明天」「后天下午五点后」）、可选**到达时间**（自然语言时间范围，如「五点后」）及车型、排序等，调用班次查询脚本；脚本内部将时间转为日期(YYYY-MM-DD)+时间段(HH:MM)，**你只负责从用户问题中提取时间表述并原样传入**，不做日期或时刻的换算。

## 1. 何时使用本 Skill

- 用户要**查火车票、列车班次、高铁、动车**，或问「某地到某地怎么坐火车」「有没高铁」等。
- 用户给出**出发地、目的地、日期**（或其一），且语境为**铁路出行**（即便未说「火车」）。
- 用户要求**按价格、出发时间、到达时间、耗时**排序或「找最便宜的」。
- 用户限定**出发时间**或**到达时间**（如「明天」「后天下午五点后」「五点后」「晚上前到」等自然语言时间范围）。

若用户明确是**飞机/航班**，应使用航班查询 skill，不要用本 skill。

## 2. 工作流程

```
用户问题（如「明天北京到上海的高铁，最便宜的」或「后天下午五点后从杭州到南京」）
    │
    ├─ 你从问题中提取：出发站、到达站、出发时间（自然语言）、到达时间（若有）、车型、排序
    ├─ 时间不做转换：原样传入脚本（如「明天」「后天下午五点后」「五点后」），由脚本内部归一为日期+时间段
    │
    ├─ 调用班次查询脚本：run_train_search.py <出发站> <到达站> <出发时间> [--arrival-time "五点后"] ...
    │       │
    │       ├─ 脚本内部：jionlp 解析时间 → 单日日期 + 时间段 → 调用火车票 API → 筛选/排序 → 输出统一 JSON
    │       └─ 输出：{ success, trains[], total_count, error? }
    │
    └─ 根据 trains 列表整理成表格或列表回复用户；失败时根据 error 说明原因
```

## 3. 你需要做的事

### 3.1 提取查询条件

从用户问题中提取并确认：

| 参数 | 说明 | 示例 |
|------|------|------|
| 出发站 from_station | 必填 | 北京、北京南、上海虹桥 |
| 到达站 to_station | 必填 | 上海、杭州东 |
| 出发时间 departure_time | 必填 | 从用户问题中**原样提取**的自然语言时间，如：明天、后天下午五点后、2025-12-21 |
| 到达时间 arrival_time | 可选 | 从用户问题中**原样提取**的时间范围，如：五点后、晚上前、下午到 |
| 车型 train_type | 可选 | 见下方**车型枚举**（固定 8 种） |
| 排序 sort_by | 可选 | 仅当用户明确说「按价格」「按时间」「最快」等时传入 |

**时间提取**：只做提取，不做换算。用户说「明天」「后天下午五点后」「3月5号上午」→ 原样作为出发时间传入；「五点后到」「晚上前到」→ 原样作为 `--arrival-time` 传入。脚本内部会用 jionlp 将自然语言转为单日日期(YYYY-MM-DD)+时间段(HH:MM)，且每次只识别**一天**的时间（如「后天下午五点后」→ 后天 17:00~23:59）。

**车型枚举（train_type，固定）**：`[G(高铁/城际), D(动车), Z(直达特快), T(特快), K(快速), O(其他), F(复兴号), S(智能动车组)]`。用户说「高铁/城际」→ G，「动车」→ D，以此类推。可传多个，如 `--train-type G --train-type D`。

**排序（sort_by）**：仅当用户明确要求时传。取值示例：`price_asc`（最便宜）、`price_desc`、`departure_asc`（尽早出发）、`departure_desc`、`arrival_asc`、`arrival_desc`、`duration_asc`（耗时最短）、`duration_desc`。

### 3.2 调用班次查询脚本

脚本位于**本 skill 根目录**下的 `scripts/run_train_search.py`（与 SKILL.md 同级的 `scripts` 子目录）。以当前环境中本 skill 的安装路径为基准拼接。

**推荐调用方式**（位置参数 + 可选键值）：

```bash
python scripts/run_train_search.py <出发站> <到达站> <出发时间> [--arrival-time "到达时间范围"] [--train-type G] [--sort-by price_asc]
```

- **出发时间**：必填，自然语言，如 `明天`、`后天下午五点后`、`2025-12-21`。脚本内部会解析为单日日期 + 时间段（仅当天，如「后天下午五点后」→ 后天 17:00~23:59）。
- **到达时间**：可选，自然语言时间范围，如 `五点后`、`晚上前`，原样传入 `--arrival-time`。

示例：

```bash
# 明天北京到上海
python scripts/run_train_search.py 北京 上海 明天

# 后天下午五点后从杭州到南京，要下午前到
python scripts/run_train_search.py 杭州 南京 "后天下午五点后" --arrival-time "下午前"

# 只要高铁，按价格从低到高
python scripts/run_train_search.py 杭州 南京 明天 --train-type G --sort-by price_asc

# 按出发时间从早到晚
python scripts/run_train_search.py 广州 深圳 "2025-12-25" --sort-by departure_asc
```

**规则**：前三个参数依次为出发站、到达站、出发时间（**自然语言，不事先转为 yyyy-mm-dd**）。可选：`--arrival-time`（自然语言时间范围）、`--train-type`、`--sort-by`。多段行程需多次调用，每次一段。

若脚本尚未实现，则根据 `docs/train-ticket-search/` 下需求与接口文档实现主入口 `run_train_search.py`，内部完成参数转化、调用火车票 API、筛选排序与统一输出。

### 3.3 根据脚本输出回复用户

- **success 为 true**：用返回的 `trains` 列表整理成表格或列表，包含车次、出发/到达站、出发/到达时刻、历时、席别与票价（若有）。**完整展示所有车次**，不要折叠或「部分省略」。
- **success 为 false**：根据 `error` 说明原因（如「未找到班次」「出发站不能为空」），并建议用户改日期、站点或条件后重试。
- **success 为 true 但 trains 为空**：说明已按条件查询但该日期/线路上暂无班次，建议改日期或线路。

## 4. 脚本输出结构（约定）

脚本标准输出为 JSON，结构约定如下（与 `docs/train-ticket-search/api.md` 一致）：

```yaml
success: true/false
trains: [ { train_no, from_station, to_station, departure_time, arrival_time, duration, seat_types[], ... }, ... ]
total_count: number
query_summary: { ... }   # 可选
error: "" 或错误说明
```

单条车次至少包含：车次号、出发站、到达站、出发时刻、到达时刻、历时；席别与票价在 `seat_types` 或等价字段中。你只需解析该 JSON 并组织成用户可读的回复，无需再调用其他接口或做二次聚合。

## 5. 参数与环境

- **火车票 API**：接口地址、请求参数、返回格式见项目内 **`docs/train-ticket-search/start.md`** 的「API接口信息」。当前 start.md 中为聚合数据航班接口示例，实现时需替换为**火车票查询接口**（如聚合数据火车票 API 等），以实际接入文档为准。
- **环境变量**：脚本若需 API Key 等，以 `docs/train-ticket-search/` 与脚本内说明为准（如 `TRAIN_API_KEY`、请求端点等）。

## 6. 执行流程小结

1. 从用户问题提取：出发站、到达站、**出发时间**（自然语言原样）、**到达时间**（若有，自然语言原样）、车型、排序。
2. 调用 `scripts/run_train_search.py <出发站> <到达站> <出发时间> [--arrival-time "…"] [--train-type …] [--sort-by …]`。不在 skill 内做日期或时刻的转换。
3. 若成功：用 `trains` 与 `total_count` 完整呈现车次列表；若失败或为空：用 `error` 说明并给出建议。

多用户同时查询时，脚本应支持并发、无拥塞；脚本内部若有多个子调用（如多数据源），应并发执行。具体实现见 `docs/train-ticket-search/architecture.md` 与 `requirements.md`。
