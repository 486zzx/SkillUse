# 周边搜索

根据用户自然语言请求，查询**目标地址周边的结果**（如餐厅、咖啡店、加油站等）：由模型提取**目标地址**与**搜索项**后，调用本 skill 提供的脚本完成查询与结果整合，输出统一格式的周边信息供你组织回复。

**推荐**：直接调用 **`scripts/run_surround_search.py`**，传入**目标地址**与**搜索项**（必选）。拿到返回的列表后，由你根据上下文以自然语言呈现给用户（表格或列表均可），**须完整展示返回的列表**，不得折叠或只显示前几条。

**重要**：传给脚本的**目标地址必须与用户输入一致**。不要补充用户没说的省、市、区、街道、门牌号，不要自行「纠正」或「补全」地址，否则可能查错地点（例如用户只说「万达广场」未提城市，若你自行补成「某市万达广场」可能查到错误城市的结果）。

---

## 1. 输入与输出

- **输入**：用户问题中应包含「目标地址」与「搜索项」（即用户要查什么，如餐厅、咖啡店、加油站）。二者均为**必选**；城市可选。
- **目标地址**：**必须原样使用用户输入的地址，不得补充、纠正或改写。** 不要写入用户没提到的省、市、区、街道、门牌号等任何信息；否则可能张冠李戴（例如用户只说「软件园」「万达广场」等，多地都有，若自行补成某省某市会查错地点）。若用户给出的是结构化地址（省+市+区+街道+门牌号），就原样传；若用户只说了简称或地标名，就只传该简称或地标名，并在回复中可提示「若您能提供更完整的地址（省+市+区+街道+门牌号），检索会更准确」。
- **搜索项**：从用户输入中提取的、**用户要搜索的内容**（如餐厅、咖啡店、加油站、酒店）。对应脚本的 `--keyword` 参数，必选。
- **输出**：脚本返回 `{ success, pois, total_count, query_summary?, error?, message? }`。成功时 `pois` 为数组，每项含 `name`、`address`、`distance`（米）、`poi_type`（类型）等。你据此整理成可读回复。
- **失败时**：`success === false` 时**必须根据 `error` 向用户说明原因**（如未找到目标地址、请求异常等）。若返回 `needs_clarification: true` 且带 `missing_params`，表示缺少必选参数，应向用户澄清并索要后再调用。
- **结果为空时**：`success === true` 且 `total_count === 0` 时，输出中会带 `message`，**须将该说明原因告知用户**（如附近暂无此类场所、可尝试更换搜索项或地址等）。

## 2. 你的职责（模型侧）

1. **提取参数**  
   从用户输入中识别：  
   - **目标地址**（必选）：**仅传递用户原话，不要自行补充或纠正。** 用户说了「北京市朝阳区三里屯路」就传该句；用户只说「万达广场」或「国贸」就只传「万达广场」或「国贸」——不要添加用户未提供的省、市、区、街道等，否则易导致检索到错误地点。可在回复中建议用户「提供更完整地址（省+市+区+街道+门牌号）可提高准确率」，但调用脚本时不得写入用户没说的内容。  
   - **搜索项**（必选）：用户要查什么，如餐厅、咖啡店、加油站、酒店；对应脚本的 `--keyword` 参数。**若用户未明确说要搜什么，不要猜测**——应先向用户澄清（如「请问您想查周边的什么？例如餐厅、咖啡店、加油站」），待用户补充后再调用脚本。  
   - **城市**（可选）：仅当**用户明确说出**城市名、或地址中**已明确包含**城市（如「北京市朝阳区…」）时，才传 `--city`。**若无法从用户输入中确定城市，不要猜测或推理**——不传 `--city` 即可（扩大检索范围），否则推断错误可能导致结果有误。

2. **参数不足时走澄清流程**  
   若用户未提供**目标地址**或**搜索项**，**不要直接调用脚本**。应先向用户说明并索要缺失项，例如：「请问要查哪里的周边？想搜什么（如餐厅、咖啡店、加油站）？」待用户补充后再调用脚本。脚本在缺少必选参数时也会返回 `needs_clarification: true` 与 `missing_params`，你可据此向用户逐项索要。

3. **调用脚本**  
   使用**命令行长参数**调用 `run_surround_search.py`，不要用 stdin 传 JSON。工作目录为 skill 根目录或 `scripts` 所在目录。

4. **呈现结果**  
   根据返回的列表与 `query_summary` 组织回复，完整列出每条结果（名称、地址、距离等），便于用户选择。

## 3. 脚本用法（run_surround_search.py）

**必选**：  
- 第 1 个位置参数：**目标地址** `location`。**必须与用户输入一致，不得自行补充省/市/区/街道/门牌号。** 用户给的是结构化地址（省+市+区+街道+门牌号）就原样传；用户只给简称或地标名就只传该简称或地标名。  
- `--keyword`：**搜索项**（用户要搜索的内容），如 `餐厅`、`咖啡店`、`加油站`。必选。

**可选**：  
- `--city`：城市名。仅当用户明确给出或地址中已包含城市名时填写；**不确定城市时不要推理**，不传此项则扩大检索范围，避免推断错误导致结果有误。

**示例**

```bash
# 用户明确说了「北京市西城区莲花池东路」时，原样传该地址；--keyword 传用户要搜的搜索项
python scripts/run_surround_search.py 北京市西城区莲花池东路118号 --city 北京 --keyword 餐厅

# 用户只说「三里屯」「国贸」等时，只传用户原话，不补省市区
python scripts/run_surround_search.py 三里屯 --keyword 咖啡店
python scripts/run_surround_search.py 国贸 --keyword 餐厅

# 用户只说「万达广场」等未带城市时，只传该句，不要自行补成「某市万达广场」
python scripts/run_surround_search.py 万达广场 --keyword 餐厅
```

**输出结构**（成功时）：  
`success: true`，`pois`: `[{ name, address, distance, poi_type, location? }]`，`total_count`，`query_summary`: `{ location, city?, keyword }`。若 `total_count === 0` 会附带 `message`，说明可能原因，须向用户转述。  
失败时：`success: false`，`error`: 原因说明，**须据此向用户说明原因**。  
缺少必选参数时：`success: false`，`needs_clarification: true`，`missing_params`: 缺失项列表（如 `["keyword"]`），`error`: 说明。

---

## 4. 流程小结

1. 用户输入 → 你提取：**目标地址**（必选，**原样传递用户所说，不补充、不纠正**）、**搜索项**（必选，用户要搜什么）、城市（可选）。**城市**仅当用户明确给出或地址中已含时才传，不确定时**不推理、不传**。  
2. **若缺少目标地址或搜索项**：先向用户澄清并索要，待用户补充后再调用脚本。  
3. 执行 `run_surround_search.py <用户原话地址> --keyword <搜索项> [--city ...]`。  
4. 根据返回的列表与 `query_summary` 组织回复，**完整展示**结果列表。
