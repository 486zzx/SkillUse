# 天气查询：需求场景与特性设计方案

基于 Open-Meteo API 全集，从用户场景出发整理需求，并给出本 skill 及扩展特性设计方案。

---

## 一、需求场景

### 1.1 场景分类

| 场景类别 | 典型表述 | 所需数据 | 对应 API |
|----------|----------|----------|----------|
| 当前天气 | 「北京现在多少度」「上海天气怎么样」 | 当前温度、体感、湿度、天气现象、风力 | Forecast current |
| 今日/明日天气 | 「今天会下雨吗」「明天北京冷不冷」 | 当日/次日最高最低温、降水、天气现象 | Forecast current + daily |
| 多日预报 | 「未来一周天气」「后天到周末」 | 多日 daily 聚合 | Forecast daily, forecast_days=7~16 |
| 逐时预报 | 「下午三点会下雨吗」「每小时温度」 | 逐时温度、降水、降水概率 | Forecast hourly |
| 穿衣/出行建议 | 「该穿什么」「适不适合出门」 | 温度、体感、降水、UV、天气现象 | Forecast current + daily（+ UV） |
| 空气质量 | 「北京空气质量」「PM2.5」 | 污染物、花粉 | Air Quality |
| 历史天气 | 「去年同期的气温」「某天有没有下雨」 | 历史逐时/逐日 | Historical Archive |
| 海洋/沿海 | 「海边浪高」「海水温度」 | 波高、海温、潮位 | Marine |
| 洪水/河流 | 「河流流量」「洪水风险」 | 河流流量 | Flood |

### 1.2 用户角色与诉求

| 角色 | 诉求 | 优先级 |
|------|------|--------|
| 普通用户 | 当前 + 短期预报、降水、穿衣建议 | P0 |
| 出行/旅行 | 多日预报、降水概率、UV | P0 |
| 农业/户外 | 降水、蒸发、土壤湿度、历史对比 | P1 |
| 健康敏感 | 空气质量、花粉、UV | P1 |
| 航海/渔业 | 海浪、海温、海流 | P2 |
| 研究/分析 | 历史长序列、多变量、多地点 | P2 |
| 应急/防灾 | 洪水、强对流相关（雷暴等） | P2 |

### 1.3 输入形态

- **地点**：城市名（中/英）、地区名、邮编；需支持模糊与中文映射（如 北京→Beijing 或经纬度）。
- **时间**：今天、明天、后天、具体日期（yyyy-mm-dd）、未来 N 天、过去 N 天、历史某段区间。
- **粒度**：当前、按日、按小时、15 分钟（仅部分区域）。
- **变量**：温度、体感、降水、天气现象、风力、UV、空气质量等；可由自然语言映射到 API 变量名。

## 二、特性设计方案

### 2.1 当前 skill 已覆盖（weather-query 现状）

| 特性 | 说明 | 实现方式 |
|------|------|----------|
| 地点解析 | 城市/地区 → 经纬度、时区 | Geocoding API + references/city_map.json（中文→英文） |
| 当前天气 | 温度、体感、湿度、天气现象、风力、降水 | Forecast current |
| 短期日预报 | 今/明/后天或未来 N 天日聚合 | Forecast daily，forecast_days 可配 |
| 自然语言日期 | 今天/明天/后天、yyyy-mm-dd | 脚本内 _normalize_date |
| 单入口脚本 | 地点 [日期] [--days N] | run_weather_search.py |
| 输出结构 | 统一 JSON：location, current, daily, requested_date | 便于模型或上游展示 |

### 2.2 扩展特性设计（按优先级）

#### P0：增强现有体验

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| F1 | 天气现象中文描述 | 所有展示场景 | 维护 WMO weather_code → 中文短句映射表（references/weather_code_map.json），脚本或模型输出时替换 code。 |
| F2 | 单位可选 | 国际化、习惯 | 脚本增加 --temperature-unit celsius|fahrenheit、--wind-unit kmh|mph 等，透传 Forecast 的 temperature_unit、wind_speed_unit。 |
| F3 | 降水概率与降水时长 | 出行、穿衣 | daily 已含 precipitation_probability_max、precipitation_hours；在 result 中保留并建议模型优先展示。 |
| F4 | UV 与防晒建议 | 户外、出行 | 请求 daily：uv_index_max 或 uv_index_clear_sky_max；在 SKILL 中约定简单分级（低/中/高/极高）与一句话建议。 |

#### P1：多日与逐时

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| F5 | 多日预报可配置天数 | 未来一周、未来两周 | 已支持 --days N；可约定默认 3、最大 16，并在 SKILL 中说明。 |
| F6 | 逐时预报子技能/子命令 | 「下午三点」「每小时」 | 新增可选模式：如 --hourly，或单独脚本 run_weather_hourly.py；请求 hourly=temperature_2m,precipitation,precipitation_probability,weather_code；按用户日期筛选后返回。 |
| F7 | 日期区间查询 | 「3月5日到3月10日」 | 解析 start_date、end_date，传 Forecast 的 start_date/end_date；返回 daily 在该区间内的切片。 |

#### P1：空气质量

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| F8 | 空气质量查询 | PM2.5、AQI、花粉 | 新脚本或子命令：run_air_quality.py 地点 [日期]；调 Air Quality API；输出 pm10、pm2_5、AQI 等，并与健康提示关联。 |
| F9 | 天气+空气质量联合 | 一次查天气与空气 | run_weather_search.py 增加 --with-air-quality，同地点先后调 Forecast 与 Air Quality，合并到同一 result。 |

#### P2：历史与海洋

| 特性 ID | 名称 | 需求场景 | 设计要点 |
|---------|------|----------|----------|
| F10 | 历史天气查询 | 去年同日、某日天气 | 新脚本：run_weather_archive.py 地点 start_date end_date [--daily/--hourly]；调 Archive API；注意 start_date/end_date 必填。 |
| F11 | 海洋天气 | 海浪、海温、潮位 | 新脚本：run_marine_weather.py 地点 [日期]；Geocoding 得经纬度后调 Marine API；cell_selection=sea；输出 wave_height、sea_surface_temperature 等。 |
| F12 | 洪水/河流流量 | 河流、洪水风险 | 新脚本或文档：Flood API 用法说明；输入经纬度或通过「河流+城市」解析得到坐标，返回 river_discharge 等。 |

### 2.3 数据与配置

| 项目 | 设计 |
|------|------|
| 中文地名 | 保持 references/city_map.json（中文→英文），优先查表再 Geocoding；可随需求扩展条目。 |
| 天气现象码 | references/weather_code_map.json：WMO code → 中文短句（如 71→「小雪」）；便于统一展示与语音。 |
| 单位与语言 | 环境变量或命令行：WEATHER_UNIT_TEMP、WEATHER_UNIT_WIND、WEATHER_LANG；脚本只负责传 API 参数与保留单位到 result。 |
| 多地点 | 当前以单地点为主；多地点可设计为多次调用或支持「地点1 地点2」在脚本内循环调用并合并 result 数组。 |

### 2.4 接口与脚本层次

```
用户 / 模型
    ↓
总入口（按场景分流）
    ├── run_weather_search.py     [已实现] 当前 + 日预报
    ├── run_weather_hourly.py     [F6] 逐时
    ├── run_weather_archive.py    [F10] 历史
    ├── run_air_quality.py        [F8] 空气质量
    ├── run_marine_weather.py     [F11] 海洋
    └── run_flood.py              [F12] 洪水（可选）
    ↓
公共层
    ├── 地理编码（Geocoding + city_map）
    ├── 日期解析（今天/明天/区间）
    └── 单位与语言配置
    ↓
Open-Meteo API 全集（见《API-接口文档》）
```

### 2.5 错误与边界

| 情况 | 处理 |
|------|------|
| 地点解析失败 | 返回明确错误（如「无法解析地点: xxx」），提示检查地名或换英文/邮编。 |
| 日期超出 API 范围 | 历史：start_date/end_date 在 1940 至今外提示；预报：forecast_days 最大 16。 |
| 海洋/洪水无数据 | 部分内陆点无海洋或河流数据，返回空或说明「该位置无海洋/河流数据」。 |
| API 限流/失败 | 记录日志，返回「天气服务暂时不可用，请稍后再试」。 |

### 2.6 与非功能需求的对应

- **可用性**：单入口、位置参数、统一 JSON，便于模型与前端解析。
- **可扩展性**：新 API（空气、海洋、历史、洪水）通过新脚本或子参数接入，复用地理与日期解析。
- **可维护性**：接口文档与需求/特性文档分离；SKILL 只写「何时用、怎么调」，细节进 references。
- **合规与引用**：非商业使用无需 KEY；若展示数据来源，可在输出或文档中注明 Open-Meteo 及所用数据源（见 API 文档末尾）。

## 三、实施优先级建议

1. **短期**：F1 天气现象中文、F2 单位可选、F4 UV——不改脚本结构，只增配置与映射表。
2. **中期**：F6 逐时、F8 空气质量、F9 天气+空气联合——新脚本或子参数，复用现有 Geocoding 与目录结构。
3. **长期**：F10 历史、F11 海洋、F12 洪水——按需实现，并在《API-接口文档》中保留完整参数说明，便于后续扩展。
