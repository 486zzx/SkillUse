# 天气查询 Skill 测试用例与评估方法

本文档描述天气查询 Skill 的**单元测试、功能测试、系统测试**用例设计及**评估方法**，便于回归与质量评估。

---

## 一、测试分层与目标

| 层级 | 目标 | 依赖 | 运行方式 |
|------|------|------|----------|
| **单元测试** | 验证单函数逻辑（解析、认证、日期、错误提取），无网络 | 无 | `pytest tests/test_weather_unit.py -v` |
| **功能测试** | 验证端到端流程与输出结构，Mock API | 无 | `pytest tests/test_weather_functional.py -v` |
| **系统测试** | 验证真实脚本子进程与可选真实 API | 可选 SENIVERSE_KEY | `pytest tests/test_weather_system.py -v` |

**推荐**：在 weatherPrj 根目录执行 `pytest tests/ -v` 跑全量测试；无 KEY 时系统测试中“真实 API”用例会自动 skip。  
**依赖**：`pip install -r .cursor/skills/weather-query/requirements-dev.txt` 或 `pip install pytest requests`。

## 五、评估方法

### 5.1 通过标准

- **单元测试**：全部通过；无 skip（除 fixture 缺失 script_cases.json）。
- **功能测试**：全部通过；无 skip。
- **系统测试**：  
  - 不依赖 KEY 的用例：全部通过。  
  - 依赖 KEY 的用例：在配置有效 KEY 的前提下通过；未配置 KEY 时 skip 不计入失败。

### 5.2 覆盖率建议

- **单元**：对 `_parse_args`、`_date_to_start`、`_extract_error`、`_get_auth_params` 做分支覆盖；目标行覆盖率 ≥ 85%（可选用 pytest-cov）。
- **功能**：主流程（仅地点、地点+日期、带 suggestion/air）与错误路径（无参数、API 错误）至少各一用例。
- **系统**：至少包含“无认证失败”“有 KEY 时一次成功调用”两类。

### 5.3 运行与报告

```bash
# 全量测试
cd weatherPrj && pytest tests/ -v

# 仅单元
pytest tests/test_weather_unit.py -v

# 带覆盖率（需 pytest-cov）
pytest tests/ -v --cov=.cursor/skills/weather-query/scripts --cov-report=term-missing
```

### 5.4 回归与 CI 建议

- 提交前本地跑 `pytest tests/ -v`。
- CI 中不注入真实 KEY，仅跑单元+功能+系统（无 KEY 部分）；有 KEY 的系统用例可在夜间或手动流水线中执行。
- 评估通过条件：单元+功能+系统（无 KEY）全部通过，且无新增 skip（除“无 KEY”导致的预期 skip）。
